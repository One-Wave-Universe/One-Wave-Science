## Node I-03: Cross-Platform Packet Intake and Reconciliation Governance

Status: GREEN (governance rule, not a physics claim — same category as I-01/I-02)

Dependencies:
Upstream: I-01 Special Rules, I-02 Node Proof/Trust Lifecycle, I-04 Scale Recurrence vs. Duplication, I-05 Active Hypothesis vs. Quarantine
Downstream: applies to all future externally-generated or cross-platform packets prior to filing

---

## Purpose

This governance rule prevents accepted, superseded, duplicated, or quarantined material from being reintroduced into the canonical One-Wave repository under new names or reorganized language.

Incoming packets must be evaluated against the live repository before they are evaluated for coherence, elegance, completeness, or apparent explanatory power.

A proposal that reads well may still be an exact duplicate, a renamed duplicate, a reconstruction of quarantined material, a rearrangement of accepted nodes, or a small unproven addition embedded inside mostly existing material.

## Governing Rule

**Check for repository matches before evaluating proposal quality.**

Quality review must not occur first. A coherent packet can appear novel when read in isolation. The first intake action is a mechanical and semantic comparison against the canonical repository.

## Mandatory Initial Trigger

Before analyzing the packet as a theory, architecture, or filing proposal:

1. Extract all proposed node names, section titles, defined terms, distinctive phrases, and claimed mechanisms.
2. Search those terms against 00_MASTER_INDEX.md, canonical node titles and definitions, dependency/cross-reference fields, superseded-material records, and quarantine/raw-material files.
3. Search both exact wording and semantic equivalents.
4. Open the matching canonical files before assessing whether the incoming packet is coherent or useful.

Renamed structures must be checked by role, relationship, dependency, and claimed function — not just by exact node-name match.

## Structural-Lineage Trigger

A quarantined-lineage review is mandatory when both conditions occur:

**Condition A — Functional similarity:** the incoming material contains substantially similar roles to a quarantined structure — a generating/imagining layer, an intermediate weighting/subconscious layer, an administrator/memory/commitment layer, or privileged access to final control/awareness/loop release.

**Condition B — Repository match:** search results identify an accepted, rejected, superseded, or quarantined node addressing the same functional structure.

When both are present, the proposal must be reconciled explicitly with the earlier material. Renaming components does not establish independence.

For the Dream Engine / M4 / Administrator packet specifically, required comparison sources: C-312, G-718, G-719, ONEWAVE_BRAIN_raw_material_UNVALIDATED.md.

## Disposition Taxonomy

Every incoming proposed node receives one primary disposition:

**1. Duplicate** — already represented by an accepted canonical node. Do not create a new node; add a cross-reference only when useful; preserve the stronger/more developed canonical formulation.

**2. Disputed or Superseded Grounding** — reconstructs a dependency, mapping, or proof route that previously failed audit. Preserve the failed grounding and identify the unresolved problem. Do not treat failure of one grounding as automatic rejection of the underlying open question.

**3. Active Hypothesis** — an unproven but open One-Wave proposal that can be clearly separated from verified claims and assigned derivation, simulation, comparison, or falsification work. File under the active hypothesis section defined by I-05.

**4. Proposed Build** — an implementable engineering architecture that can be tested without claiming that nature or consciousness already uses it. File in the appropriate proposed-build book.

**5. Quarantined Lineage** — use only when the source itself must remain outside active canon because of contradiction, corrupted or unclear provenance, or a superseded formulation that cannot be safely treated as current. Quarantine must not be assigned solely because a question remains unknown.

**6. Unproven Delta** — contains a specific claim not already present, not yet derived/measured/simulated/grounded. Extract the exact new claim; separate it from duplicated surrounding material; state what evidence or derivation would be required for promotion.

A packet may contain multiple categories across different proposals; each proposal receives its own primary disposition and any useful secondary tags.

## Required Intake Sequence

```
1. Identify source and date
2. Extract proposed claims and node names
3. Search the Master Index
4. Search canonical definitions
5. Search superseded and quarantined material
6. Open and compare matching files
7. Classify each proposal
8. Extract any genuine new delta
9. Apply grounding and promotion tests
10. Decide whether to reject, cross-reference, preserve as active hypothesis, route as proposed build, quarantine, revise, or file
```

Prohibited sequence: read for coherence → become persuaded by the architecture → prepare nodes → check the repository afterward.

## Reconciliation Record

Each intake packet produces a reconciliation table: Incoming proposal | Canonical match | Quarantined/superseded ancestor | Exact new delta | Disposition | Required action.

Retained even when no new canonical nodes are created. A successful rejection of duplicates, repair of failed grounding, or routing into active hypothesis/proposed-build status is an audit result, not a failed filing task.

## Promotion Requirement

Material classified as Active Hypothesis, Proposed Build, Quarantined Lineage, or Unproven Delta may be promoted into accepted canon only when it provides something not supplied by the rejected or existing formulation: derivation from accepted nodes, a demonstrably distinct functional architecture, explicit hardware/software implementation requirements, measurable output differences, falsifiable predictions, simulation results, experimental evidence, or direct resolution of the grounding objections recorded in the earlier audit (G-719).

New terminology, improved prose, additional diagrams, or broader analogy do not satisfy the promotion requirement by themselves.

## Attribution Rule

Repository history must describe process failures in attributable, impersonal language. Do not include unattributed first-person statements. Where known, record the source platform/system, date received, importing workflow, reviewer/audit process, and repository version used for reconciliation. No system may claim personal responsibility for an action performed by another, unidentified system.

## Final Governance Principle

Canonical novelty is determined by comparison with the live repository, not by how new, coherent, or persuasive an incoming packet appears when read alone.
